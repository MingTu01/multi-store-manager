# Multi Shop Link - 升级包管理规范

## 一、版本号规则

### 1.1 版本格式
```
v主版本.次版本.修订号
```

- **主版本（Major）**：重大架构变更或不兼容更新，由管理员手动指定
- **次版本（Minor）**：新功能、功能增强，**每次更新必须递增**
- **修订号（Patch）**：Bug 修复、小改动，**每次更新必须递增**

### 1.2 版本号递增规则

| 场景 | 版本变化 | 示例 |
|------|----------|------|
| 新功能 | Minor +1，Patch 归零 | v1.0.2 → v1.1.0 |
| Bug 修复 | Patch +1 | v1.0.2 → v1.0.3 |
| 重大更新 | Major +1，Minor/Patch 归零 | v1.0.2 → v2.0.0 |

### 1.3 版本号存储位置

每次更新必须同步修改以下 **4 个文件**：
1. `apps/server/package.json` — 服务端版本
2. `apps/web/package.json` — 前端版本
3. `apps/server/data/version.json` — 系统运行时读取的版本
4. `package.json`（部署仓库）— 部署仓库版本

## 二、升级包规范

### 2.1 升级包类型

**只使用升级包，不使用全量包。**

| 类型 | 说明 | 是否使用 |
|------|------|----------|
| 升级包 | 只包含代码变更 | ✓ 推荐 |
| 全量包 | 包含所有文件（含 OCR 模型等） | ✗ 仅首次部署使用 |

### 2.2 升级包命名规则

```
multi-shop-link-upgrade-v{版本号}.zip
```

示例：`multi-shop-link-upgrade-v1.1.16.zip`

### 2.3 升级包内容（必须严格按此结构）

```
multi-shop-link-upgrade-vX.X.X.zip
├── package.json                    # 根目录版本信息
├── version.json                    # 版本元数据
├── server-src/                     # 服务端源码
│   ├── index.ts
│   ├── db.ts
│   ├── auth.ts
│   ├── routes/
│   ├── middleware/
│   └── ...
└── web-dist/                       # 前端构建产物
    ├── index.html
    ├── assets/
    └── ...
```

### 2.4 升级包排除项

以下文件 **不包含** 在升级包中：
- `node_modules/` — 依赖包
- `data/` — 数据库文件（用户数据）
- `uploads/` — 上传的文件（用户数据）
- `backups/` — 备份文件
- `*.traineddata` — OCR 模型文件（2MB+，极少变更）

### 2.5 升级包大小参考

| 包类型 | 典型大小 | 说明 |
|--------|----------|------|
| 升级包 | 0.5-3 MB | 只含代码变更 |
| 全量包 | 8-10 MB | 含 OCR 模型等（仅首次部署） |

## 三、升级流程

### 3.1 ZIP 升级

1. 管理员登录 → 系统设置 → 系统升级
2. 选择 ZIP 升级 → 上传升级包
3. 系统验证升级包（版本号、文件完整性）
4. 点击「开始升级」→ 二次确认
5. 上传进度条显示上传速度
6. 系统自动执行：
   - 备份数据库（自动标记为"升级前"）
   - 解压升级包
   - 更新版本信息
   - 覆盖系统文件
   - 重启服务
7. 升级完成后点击「确认并刷新页面」

### 3.2 在线更新

1. 管理员登录 → 系统设置 → 系统升级
2. 系统每 30 秒自动检查更新（也可手动点击「检查更新」）
3. 发现新版本后点击「执行更新」
4. 系统自动完成：备份 → 下载 → 覆盖 → 重启
5. 升级进度实时显示
6. 完成后自动刷新页面

### 3.3 Docker 环境注意事项

在 Docker（1Panel）环境下：
- `process.exit(0)` 会导致容器重启
- 升级包文件会覆盖到容器内对应路径
- 如果容器使用 volume 挂载，文件变更会持久化
- 如果没有挂载，容器重启后文件会丢失 → 需要重建镜像

**推荐方案：** 将 `data/`、`uploads/`、`backups/` 通过 volume 挂载。

## 四、回滚方案

### 4.1 自动备份

升级前系统会自动创建备份：
```
backups/pre-upgrade-{时间}.zip
```

### 4.2 手动回滚

如果升级失败：
1. 系统设置 → 数据备份
2. 找到升级前的备份文件
3. 点击「恢复」
4. 确认恢复
5. 系统自动重启

## 五、常见问题

| 问题 | 解决方案 |
|------|----------|
| 升级包上传失败 | 检查网络连接，确保 ZIP 文件完整 |
| 升级后版本号没有变化 | 检查升级包中 package.json 版本号是否正确 |
| 升级后页面白屏 | 清除浏览器缓存，或使用无痕模式访问 |
| 升级后运行时间没有重置 | 说明服务器没有重启，Docker 环境下需要手动重启容器 |
| 数据库会丢失吗 | 不会。升级包不包含数据库文件，升级前会自动备份数据库 |
| 可以跨版本升级吗 | 可以。系统会直接覆盖文件，不依赖版本连续性 |
